# 评分解释页面重构方案

## 1. 背景

当前智能调度的“评分变更记录”弹窗承载了太多职责：评分结论、评分明细、运行态信息、公式说明、调试上下文和日志审计混在一个视图里。随着新版评分引擎改为一级评分项和评分事件追踪，继续在旧弹窗里补字段会让页面更难理解。

本方案将该页面重构为“评分解释中心”，用新版评分引擎的一套 `ScoreItem` 和 `ScoreEvent` 数据驱动展示，按“结论 -> 原因 -> 证据 -> 建议 -> 技术细节”的顺序组织信息。

## 2. 页面目标

- 3 秒内看懂这次评分为什么涨跌。
- 30 秒内定位贡献最大的评分项。
- 运营能判断是否需要干预渠道。
- 技术能追踪样本来源、窗口、公式和 before/after/delta。
- 避免前端另写一套评分推导逻辑，页面解释必须来自后端评分事件。

页面核心回答四个问题：

```text
这次评分为什么变了？
哪些评分项贡献最大？
当前渠道还能不能调度？
运营或技术下一步要不要处理？
```

## 3. 页面定位

该页面只解释一次评分事件或当前 runtime 的最近一次评分状态，不承担完整运行态详情、调度详情、日志转储和公式文档职责。

默认展示：

- 结论
- 分数变化
- 主要原因
- 样本来源
- 一级评分项明细
- 运营建议

默认折叠：

- runtime key
- capability fingerprint
- formula 原始表达式
- request id / trace id
- score event JSON
- 原始上下文

## 4. 信息架构

### 4.1 顶部对象与结论区

示例：

```text
toito / gpt-5.5 / codex-plus
当前评分 0.973
本次变化 +0.083
状态 可调度
样本数 3,606
最近更新 2026-05-27 00:09:32
```

一句话解释：

```text
本次评分上升，主要来自首包速度改善、上游错误率下降、完成率提升。当前无阻断风险，可继续参与调度。
```

### 4.2 核心指标卡

第一屏只保留最重要的卡片：

- 当前评分
- 上次评分
- 本次变化
- 调度状态
- 样本可信度

样本可信度根据样本数、样本来源和缺失项计算展示，不作为新的评分项。

### 4.3 贡献解释区

分为主要加分项和主要扣分项。

示例：

```text
主要加分项
首包延迟 +0.345
上游错误率 +0.168
完成率 +0.148

主要扣分项
成本 -0.052
队列压力 -0.031
```

没有扣分时展示：

```text
暂无明显扣分项
```

贡献项必须直接来自 `ScoreItem.delta` 或后端返回的 `top_positive_items` / `top_negative_items`，前端不再自行推断。

### 4.4 运营建议区

运营建议用于把评分结果翻译成动作。

建议枚举：

- 可继续观察
- 建议排查上游错误
- 建议关注首包变慢
- 建议检查成本配置
- 样本不足，暂不判断
- 恢复中，等待更多探活样本
- 建议临时降权或隔离

建议由后端根据评分项、阻断状态和样本来源生成，前端只负责展示。

### 4.5 一级评分项明细表

表格列：

```text
评分项 | 当前值 | 得分 | 权重 | 贡献 | 本次变化 | 样本窗口 | 来源 | 原因
```

评分项固定来自新版评分引擎：

- 完成率
- 上游错误率
- 首包延迟
- 完整耗时
- 吞吐速度
- 空输出率
- 流中断率
- 并发负载
- 队列压力
- 首包积压
- 成本
- 分组优先级

不再展示旧聚合概念：

- 健康分
- 体验分
- 输出质量分
- 动态速度分

缺失样本项展示为灰色 `样本不足`，不展示假低分。

### 4.6 评分事件时间线

时间线用于追踪最近几次评分变化。

每条事件展示：

```text
0.890 -> 0.973  +0.083
来源：真实请求
主要变化：首包延迟 +0.050，上游错误率 +0.096，完成率 +0.129
时间：2026-05-27 00:09:32
```

探活事件必须明确标记，避免运营误以为真实请求已经完全恢复。

## 5. 样本来源

每次评分事件必须标记来源：

- 真实请求
- 探活请求
- 首包前失败
- 中途流中断
- 空输出
- 跳过评分

探活样本可以影响评分，但 UI 必须显示 `探活样本` 标签。动态价格统计和真实用户成功率仍不使用探活样本。

跳过评分时展示跳过原因：

- 用户主动取消
- 余额不足
- 本地并发限制
- 未进入智能调度链路失败
- 配置错误隔离跳过

## 6. 分数口径

页面需要明确区分两个口径：

```text
稳定评分：长期健康评分，用于渠道健康趋势和恢复判断。
本次调度评分：叠加当前并发、队列、首包积压后的实时选路分。
```

评分变更默认展示稳定评分变化。本次调度评分只作为辅助信息展示，避免用户把实时压力导致的选路分波动理解成渠道健康突然变差。

## 7. 后端接口建议

后端直接返回页面所需结构，前端不自行推导主要原因。

```json
{
  "summary": {
    "title": "本次评分上升",
    "description": "主要来自首包速度改善、上游错误率下降、完成率提升。",
    "status": "available",
    "recommendation": "可继续观察"
  },
  "before_total": 0.89,
  "after_total": 0.973,
  "delta": 0.083,
  "score_type": "stable",
  "sample_source": "real_request",
  "sample_count": 3606,
  "top_positive_items": [],
  "top_negative_items": [],
  "score_items": [],
  "timeline": [],
  "debug_context": {}
}
```

### 7.1 summary

```text
title
description
status
recommendation
risk_level
```

### 7.2 score_items

复用新版评分引擎 `ScoreItem`：

```text
key
name
category
raw_value
window
score
weight
weighted_score
previous_score
delta
sample_count
missing_reason
formula
reason
source
```

### 7.3 timeline

```text
event_id
created_at
before_total
after_total
delta
sample_source
changed_items
request_id
trace_id
is_health_probe
```

## 8. UI 原则

第一屏只看结论，不看 JSON。

视觉层级：

```text
结论卡
核心指标卡
加分/扣分贡献卡
一级评分项表格
评分事件时间线
技术详情折叠区
```

颜色规则：

- 加分：绿色
- 扣分：红色
- 缺失样本：灰色
- 可调度：青绿
- 恢复中：蓝色
- 阻断/异常：橙红

数字格式：

- 评分：`0.973`
- 变化：`+0.083`
- 权重：`9.1%`
- 延迟：`4.21s`
- 样本：`3,606`

## 9. 交互细节

- 默认只展示 Top 3 加分项和 Top 3 扣分项。
- `查看全部评分项` 展开完整一级评分项表格。
- 技术字段默认折叠，通过 `技术详情` 展开。
- 长 runtime key 显示为摘要，提供复制按钮。
- score event JSON 只在技术详情中展示。
- 点击评分项可展开该项的公式、窗口和样本说明。
- 点击时间线事件可切换下方明细表到该事件。

## 10. 产品、运营、技术分工

### 产品

- 确认页面定位为评分解释，不做日志转储页。
- 定义可调度、恢复中、阻断、样本不足等状态文案。
- 明确稳定评分和本次调度评分的解释边界。

### 运营

- 使用运营建议判断是否需要干预。
- 重点关注扣分项、探活标签、样本不足和异常状态。
- 不需要阅读 formula 或 JSON 即可完成基本判断。

### 技术

- 后端统一从 `ScoreEvent` 和 `ScoreItem` 输出页面数据。
- 前端不重复实现评分原因推导。
- 保留 debug context 供排查，但默认折叠。
- 确保探活样本、真实请求样本、跳过评分样本口径一致。

## 11. 落地阶段

### P0：信息层级重构

- 弹窗标题改为 `评分解释`。
- 顶部改为结论卡和核心指标卡。
- 隐藏长 runtime key 和原始 JSON。

### P1：接入新版评分项

- 使用 `ScoreItem` 渲染一级评分项表格。
- 移除健康分、体验分、输出质量分、动态速度分等旧聚合展示。
- 缺失样本展示 `样本不足`。

### P2：贡献和建议

- 展示 Top 加分项和 Top 扣分项。
- 展示样本来源和运营建议。
- 探活样本增加醒目标记。

### P3：评分事件时间线

- 展示 before/after/delta。
- 支持查看最近评分变化。
- 支持点击事件切换明细。

### P4：技术详情

- 折叠展示 runtime key、formula、request id、trace id、score event JSON。
- 增加复制和排查入口。

## 12. 验收标准

- 第一屏能看出当前评分、变化方向、主要原因和调度状态。
- 所有评分项来自后端 `ScoreItem`，前端不自行计算主要原因。
- 探活样本、真实请求、跳过评分有明确标签。
- 成本、速度、成功率、流中断、空输出等一级项能独立展示变化。
- 技术详情默认不干扰运营使用。
- 页面不再出现健康分、体验分、动态速度分等旧聚合概念。
