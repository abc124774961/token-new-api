# 用户请求视图 UI 设计

## 目标

`用户请求视图` 是模型网关观测页的默认主视图，统计口径面向用户请求最终结果，而不是内部渠道 attempt。页面只展示主内容区域，不调整顶部导航栏、侧边栏和全局布局。

## UI 结构

- 顶部 Tab 顺序：`用户请求视图 / 运营视图 / 工程视图`。
- 左侧大卡：`用户感知健康`，展示健康分、请求成功、最终失败、自动恢复。
- 右侧 2x3 KPI：用户成功率、用户请求数、自动恢复、平均响应、P95 响应、最终失败。
- 中部趋势图：用户成功率、请求量、P95 响应。
- 下方双列表：模型用户体验排行、分组用户体验排行。
- 底部表格：最近用户请求，列为完成时间、请求 ID、模型/分组、最终状态、响应耗时、自动恢复。

## 数据口径

- 单位是 `request_id`，每个用户请求只计一次。
- 内部失败后最终成功：`final_success=true`，`recovered=true`。
- 多次失败最终失败：只计 1 次最终失败。
- 最终失败仅保留泛化类型：`rate_limit / timeout / upstream_error / stream_interrupted / server_error`。
- 查询只读取 `model_gateway_user_request_summaries` 预聚合表，不临时扫描明细表做 `request_id` group。

## 不展示边界

用户请求视图不展示渠道列、Replay 操作、调度详情、中间错误链路、候选渠道、候选模型和具体上游错误消息。运营视图和工程视图继续保留这些排障信息。

## 性能要求

- 写入链路异步 upsert 预聚合表，避免请求路径阻塞。
- summary 查询按时间窗口、模型、分组和请求 ID 走预聚合表索引。
- 页面仍使用现有 socket snapshot/delta 和短缓存策略；用户视图依赖预聚合后的最终态，避免将正在重试的 429 中间态展示成最终失败。
- recent、趋势、排行使用 `scan_limit` 上限保护，避免大窗口全表扫描拖慢首屏。

## 视觉基线

延续 classic 主题的 AI 运维控制台风格：浅色玻璃卡片、teal/cyan 主色、紧凑指标卡、低噪声表格和适合扫描的密集布局。预览图见 `docs/model-gateway/user-request-view-ui-preview.png`。
